Documentation Resources

1. Why Document

Documentations helps us to understand the code we write. If working on open source library, users and contributors will have a higher tendency to use yor library in their projects. End users of documentation include: You, co-workers, your users

2. Anatomy to READMEs

  • Title
  • Description: Capture spirit of project clearly and precisely. Include logo if it has one.
  • Include information absolutely necessary for understanding code. Includes getting started/installation, usage, contributing, license. Can be short or long, depends on project. All the essentials. Don't make assumptions on what the end user knows.
    • What steps need to be taken?
    • What should the user already have installed or configured?
    • What might they have a hard time understanding right away?

3. Documentating a Growing Codebase

  • Known bugs
  • FAQ
  • Table of contents

4. Markdown Syntax

  • double asterisks ** ** makes text bold
  • underscores _ _ makes text italicized
  • backticks ` ` makes an inline code (not a regular word)
  • # before text creates a heading tag (h1 to h6) depending on how many hashes used
    • ## = h2
    • ### = h3
  • ~~ creates a strikethrough
  • > quotes text
  • Lists can be made with - or * (ul list) or 1.2.3.(ol)
  • Links can be made using [] and then (). Example: [GitHub Pages](URL)
  • Tasks lists can be created using [ ] and [x]
  • Pargraphs - leave blank line between lines of text
  • ``` ``` used for codeblock
  • README files should be created as .md files (md = markdown)
  • Markdown files cannot be opened in browser like HTML, but can preview using Dillinger
  • In atom, Markdown preview is baked right into the program (in the 'Packages' menu)
